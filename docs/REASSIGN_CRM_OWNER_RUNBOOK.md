# Reassign CRM owner

Разовая ручная передача сущностей Bitrix от одного ответственного к другому.

Главный безопасный сценарий — работать от сделок. Тогда `max_total` считается как лимит выбранных сделок, а связанные компании и контакты подтягиваются дополнительно и не расходуют лимит.

## Базовый порядок

1. Запустить с `dry_run=true`.
2. Скачать artifact `reassign_crm_owner_log.csv`.
3. Проверить список сделок, компаний, контактов, старого и нового ответственного, целевую стадию и причину неудачи.
4. Если всё верно — повторить запуск с `dry_run=false`.

## Основные параметры

- `source_user_id` — старый ответственный.
- `target_user_id` — новый ответственный.
- `include_deals=true` — основной режим переноса заявок/сделок.
- `include_companies=true` — вместе со сделками переносить связанные компании.
- `include_contacts=true` — вместе со сделками переносить связанные контакты.
- `include_leads=false` — лиды обычно не трогать, потому что текущий контур работает через компании + сделки.
- `deal_category_id` — воронка сделок. `all` = все воронки.
- `deal_stage_ids` — исходные стадии, из которых забирать сделки. Например `NEW,EXECUTING`. `all` = все стадии.
- `include_closed_deals` — переносить закрытые сделки. Для неудачных сделок обычно нужно `true`, если они уже закрыты.
- `target_deal_stage_id` — новая стадия для выбранных сделок. Пусто = оставить текущую стадию.
- `failure_reason_id` — ID причины неудачи из списка ниже.
- `failure_reason_field` — поле причины неудачи. Если выбран `failure_reason_id` и поле пустое, код использует `UF_CRM_1779448756033`.
- `failure_reason_text` — редкое ручное значение вместо `failure_reason_id`. Обычно оставить пустым.
- `max_total` — если `include_deals=true`, это лимит выбранных сделок. Связанные компании и контакты не считаются.
- `limit` — лимит на каждый тип сущности. Обычно можно оставить `0` и управлять через `max_total`.

## Причины неудачи

| CloseReasonID | CloseReasonName |
|---:|---|
| 400 | Дубль сделки |
| 394 | Клиент отказался |
| 402 | Не ведёт деятельность / компания неактивна |
| 386 | Не дозвонились |
| 396 | Не подходит по критериям |
| 392 | Нет данных организации |
| 398 | Ошибка данных |
| 404 | Проиграли аукцион |
| 390 | Уже работает с Евразией |
| 388 | Уже работает с конкурентом |

## Пример: передать 10 новых сделок другому менеджеру

- `dry_run=true`
- `source_user_id` — текущий менеджер
- `target_user_id` — новый менеджер
- `include_deals=true`
- `include_companies=true`
- `include_contacts=true`
- `include_leads=false`
- `deal_stage_ids=NEW`
- `target_deal_stage_id=` пусто
- `failure_reason_id=none`
- `max_total=10`

После проверки CSV-лога повторить с `dry_run=false`.

## Админский сценарий: перекинуть неудачные сделки

Используется, если нужно забрать сделки у одного менеджера, передать другому, поставить нужную стадию и массово указать причину неудачи.

Пример:

- `dry_run=true`
- `source_user_id` — менеджер, у которого забираем
- `target_user_id` — менеджер, которому передаём
- `include_deals=true`
- `include_companies=true`, если вместе со сделками надо передать компании
- `include_contacts=true`, если надо передать контакты
- `include_closed_deals=true`, если сделки уже закрыты
- `deal_stage_ids=LOSE` или конкретные исходные стадии
- `target_deal_stage_id=LOSE` или другая нужная стадия воронки
- `failure_reason_id=394`, если причина «Клиент отказался»
- `failure_reason_field=` пусто, если используется стандартное поле `UF_CRM_1779448756033`
- `max_total=20`

Сначала CSV-лог, потом запись. Никаких геройских массовых запусков вслепую: Bitrix не Excel, Ctrl+Z там не живёт.

## Что попадёт в CSV-лог

В логе будут:

- тип сущности;
- ID сущности;
- название;
- старый и новый ответственный;
- текущая стадия;
- целевая стадия;
- поле причины неудачи;
- `failure_reason_id`;
- название причины;
- значение, которое будет записано в Bitrix;
- действие: `dry_run_update`, `updated`, `skip_*` или `update_error`.
