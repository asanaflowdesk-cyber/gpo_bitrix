# CONFIG STRUCTURE

## 1. Менеджеры

Основной справочник менеджеров:

```text
eqazyna_bitrix/config/managers.yml
```

Там хранятся активные менеджеры, их Bitrix ID, имена, филиалы и технические source-пользователи.

Чтобы временно убрать менеджера из автоматического распределения, не удаляй строку. Поставь:

```yaml
active: false
```

Технические пользователи 36/44 используются только как источник старых/неразобранных карточек. Они не считаются нормальными менеджерами и не могут быть историческим владельцем пакета.

## 2. Основная логика автоматического распределения

Автоматический pipeline не использует ручные YAML-закрепления руководителей и не использует BIN-закрепления как приоритет.

Порядок такой:

1. существующая сделка по `ORIGIN_ID = application_key` → пропуск без обновления и без повторного eGov-обогащения;
2. новая сделка → поиск исторического менеджера по руководителю/учредителю;
3. самый старый нормальный deal этого руководителя;
4. если deal нет — самая старая компания;
5. если компании нет — контакт руководителя;
6. если истории нет — менеджер с минимальной активной нагрузкой по e-Qazyna-сделкам;
7. мягкий лимит новых назначений — 30 сделок только в стадиях `Новая` + `В работе`; остальные стадии в лимит не входят; исторические пакеты лимит игнорируют.


### Стадии для лимита

Лимит 30 не считается по всем открытым сделкам. В расчёт попадают только сделки в стадиях `Новая` и `В работе`.

Параметр:

```text
BITRIX_ASSIGNMENT_LOAD_STAGE_IDS=ALL
```

По умолчанию `ALL` означает все незакрытые сделки e-Qazyna: новая, в работе, согласование, сбор документов и любые другие не финальные стадии. Если нужен более узкий расчет, можно указать конкретные технические `STAGE_ID` через запятую, например: `C2:NEW,C2:UC_WORK`.

## 3. Админское ручное переназначение

Основной ручной инструмент:

```text
.github/workflows/reassign-crm-owner.yml
eqazyna_bitrix/reassign_crm_owner.py
```

Он позволяет выбрать:

```text
source_user_id
target_user_id
deal_stage_ids
target_deal_stage_id
failure_reason_id
failure_reason_field
failure_reason_text
include_companies / include_contacts / include_deals
```

Сначала запускать `dry_run=true`, потом проверять CSV artifact, только затем `dry_run=false`.

## 4. Файлы, которые не надо править руками

```text
eqazyna_bitrix/pipeline.py
eqazyna_bitrix/distribute_companies.py
eqazyna_bitrix/audit_repair_deal_packages.py
eqazyna_bitrix/audit_repair_director_package_owners.py
```

Бизнес-правила должны меняться через конфиги менеджеров и параметры workflow, а не через ручное редактирование кода.

## Admin reassignment close reasons

Workflow `Reassign CRM owner` supports predefined Bitrix close-reason enum IDs:

- `400` — Дубль сделки
- `394` — Клиент отказался
- `402` — Не ведёт деятельность / компания неактивна
- `386` — Не дозвонились
- `396` — Не подходит по критериям
- `392` — Нет данных организации
- `398` — Ошибка данных
- `404` — Проиграли аукцион
- `390` — Уже работает с Евразией
- `388` — Уже работает с конкурентом

If `failure_reason_id` is selected and `failure_reason_field` is empty, the script writes the value to `UF_CRM_1779448756033`.
