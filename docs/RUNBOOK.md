# RUNBOOK

## Проверить Bitrix

Actions → Bitrix diagnostics → Run workflow.

Ожидаемый результат: artifact `bitrix-diagnostics` с JSON-файлами.

## Проверить eGov

Отдельного eGov workflow в архиве нет. Проверка eGov идёт через основной поток в `dry_run=true`: если eGov вернёт ошибку, она попадёт в Excel/JSON-лог по строкам. Для временной изоляции проблемы можно запустить основной поток с `no_egov=true`: тогда проверяется e-Qazyna + Bitrix без обращения к eGov.

## Запустить основной поток

Основной код не содержит собственного расписания. Регулярный запуск идёт внешним cron schedule site.

Ручной запуск через GitHub Actions остаётся для проверки:

```text
Actions → e-Qazyna to Bitrix CRM AUTO → Run workflow
```

Для безопасной проверки:

```text
push_bitrix = true
dry_run = true
```

Для записи в CRM:

```text
push_bitrix = true
dry_run = false
```

## Что считается нормальным результатом

В логе могут быть действия:

```text
created_company_created_deal
existing_company_created_deal
existing_deal_skipped
dry_run_company_and_deal
dry_run_create_deal
error
```

`existing_deal_skipped` — не ошибка. Это защита от повторной обработки: если сделка с таким `ORIGIN_ID = application_key` уже есть, код ничего не обновляет и переходит к следующей заявке. При запуске с Bitrix-записью такие заявки отсекаются до eGov, чтобы не гонять eGov повторно каждые 5 минут.

## Основное правило распределения

1. Если сделка уже есть — пропустить без изменений.
2. Если сделка новая — найти руководителя/учредителя.
3. Если по руководителю уже есть история — взять самого старого нормального ответственного: сначала из сделки, потом из компании, потом из контакта.
4. Исторический руководитель сильнее лимитов и любых старых/ручных закреплений.
5. Если руководитель полностью новый — назначить менеджера с минимальной активной нагрузкой по e-Qazyna-сделкам.
6. Мягкий лимит для новых руководителей — 30 сделок только в стадиях `Новая` + `В работе`. Остальные стадии в лимит не входят. Если все выше лимита, выбирается менеджер с минимальной нагрузкой.
7. Список стадий, которые входят в лимит, задаётся параметром `assignment_load_stage_ids` / переменной `BITRIX_ASSIGNMENT_LOAD_STAGE_IDS`. По умолчанию: `NEW,EXECUTING`; для твоей воронки надо указать фактические `STAGE_ID` стадий `Новая` и `В работе`.
8. Технические пользователи 36/44 не считаются нормальными менеджерами.

## Если надо вручную перекинуть сделки

Использовать только админский workflow:

```text
Actions → Reassign CRM owner → Run workflow
```

Сначала `dry_run=true`, проверить CSV-лог, потом `dry_run=false`.

Админский workflow умеет:

```text
source_user_id → target_user_id
deal_stage_ids = какие стадии забрать
target_deal_stage_id = в какую стадию поставить выбранные сделки
failure_reason_id = причина неудачи из списка Bitrix
failure_reason_field = поле причины неудачи, по умолчанию UF_CRM_1779448756033
failure_reason_text = редкое ручное значение вместо failure_reason_id
```

Это отдельный ручной режим. Он не должен смешиваться с автоматическим распределением.
